# Khoj Agent System

## Agent Model

Agents in Khoj are specialized AI assistants with configurable personalities, tools, knowledge bases, and chat models. They are stored as Django `Agent` model instances and can be shared across users on a server.

### Agent Properties

- **Name**: Display name for the agent
- **Persona/Personality**: System prompt text defining the agent's behavior and style
- **Chat Model**: Which LLM to use for this agent (can differ from default)
- **Tools**: Which tools the agent can access (search, online search, code execution, computer use)
- **Knowledge Base**: Dedicated document collection (uploaded during agent creation)
- **Public/Private**: Whether the agent is accessible to all server users
- **Avatar**: Visual identity for the agent

### Agent Creation

Two paths:
1. **Admin panel** (`/server/admin/database/agent/`): Full control over all properties
2. **Web UI** (`/agents`): User-facing agent creation with document upload

When creating via the web UI, users upload documents that become the agent's dedicated knowledge base. The agent's persona defines its role and behavior through a system prompt that gets injected into every conversation.

## Tool System

### Available Tools

Khoj agents have access to a fixed set of built-in tools (not an extensible plugin system):

1. **Knowledge Base Search** -- Semantic search over the user's indexed documents
   - Bi-encoder retrieval + cross-encoder reranking
   - Filtered by the agent's knowledge base if agent-specific docs exist
   - Source: `src/khoj/search_type/text_search.py`

2. **Online Search** -- Web search via SearxNG
   - Agent determines when fresh information is needed
   - Results are scraped and summarized
   - Source: `src/khoj/processor/tools/online_search.py`

3. **Code Execution** -- Python code generation and execution
   - Runs in Terrarium sandbox (Pyodide in Docker)
   - Can generate plots, do data analysis, compute results
   - Alternative: E2B cloud sandbox
   - Source: `src/khoj/processor/tools/run_code.py`

4. **Computer Use/Operator** -- Desktop automation via VNC
   - Optional service (requires `KHOJ_OPERATOR_ENABLED`)
   - VNC-accessible desktop environment
   - Agent can interact with desktop applications
   - Source: `src/khoj/processor/tools/operator.py`

### Tool Selection (Intent Detection)

Tool selection is NOT hardcoded. The LLM itself decides which tools to use:

1. User sends a message
2. Khoj sends the message + available tool descriptions to the LLM (intent detection call)
3. LLM returns structured output indicating: which tools to invoke, what queries to use
4. Khoj executes the selected tools
5. Tool results are assembled into context for the final response

This is effectively function calling / tool use, but implemented through prompt engineering rather than native function calling APIs. The intent detection prompt describes available tools and asks the model to select appropriate ones.

### Tool Execution Flow

```
Intent Detection (LLM call)
    │
    ├── tools_needed: ["knowledge_search", "online_search"]
    ├── inferred_queries: ["quarterly revenue 2025", "market trends AI"]
    │
    ▼
Parallel Tool Execution
    ├── text_search.search() → document chunks
    ├── online_search.search() → web results
    │
    ▼
Context Assembly
    │ Merge all tool results + conversation history
    ▼
Final LLM Response (with full context)
```

## Automations

Automations are scheduled, recurring agent tasks. Think of them as cron jobs powered by agents.

### How Automations Work

1. User creates an automation with a natural language query (e.g., "Summarize AI news every morning")
2. Khoj uses the LLM to convert this into:
   - A cron schedule (e.g., `0 8 * * *`)
   - A cleaned query to execute
   - A subject line for the email
3. At the scheduled time, Khoj runs the query through the full chat pipeline
4. Results are emailed to the user

### Automation Properties

- **Query**: The prompt to execute
- **Cron schedule**: When to run (generated by LLM from natural language)
- **Timezone**: User's timezone for scheduling
- **Subject**: Email subject line
- **Agent**: Which agent to use (optional)

### Load Distribution
Khoj randomizes the minute in cron schedules to distribute load, avoiding thundering herd when many automations share the same schedule.

## Research Mode as Agent Behavior

Research mode (`/research`) transforms the agent into an iterative researcher:

1. **Plan**: Agent creates a research plan with sub-questions
2. **Iterate**: For each sub-question:
   - Search knowledge base and/or web
   - Execute code if needed
   - Evaluate if the sub-question is answered
3. **Synthesize**: Combine all findings into a comprehensive answer
4. **Cite**: Include references to all sources used

Each iteration is tracked as a `ResearchIteration` object with:
- Queries executed
- Results obtained
- Reasoning/analysis
- Whether the sub-question was resolved

This achieved a 141% accuracy improvement over baseline LLM performance on the FRAMES benchmark.

## Agent-to-Knowledge-Base Interaction

### Per-Agent Knowledge Bases
Each agent can have its own document collection, separate from the user's main knowledge base:
- Documents uploaded during agent creation
- Indexed and embedded like any other content
- Search is scoped to the agent's collection when chatting with that agent

### Shared Knowledge Base Access
Agents also have access to the user's general knowledge base (unless restricted). The search combines:
- Agent-specific documents (higher priority)
- User's general document index

## Key Design Patterns

1. **Agent = persona + tools + knowledge**: Simple but effective composition model
2. **LLM-driven tool selection**: No hardcoded routing rules; the model decides
3. **Fixed tool set**: Not a plugin architecture; tools are built-in capabilities
4. **Automation = scheduled agent**: Reuses the full chat pipeline for recurring tasks
5. **Research mode = iterative agent**: Multi-step reasoning as a first-class mode
6. **Email delivery**: Automations deliver results via email, not in-app notifications
